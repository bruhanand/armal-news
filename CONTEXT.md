# Armal News

A TikTok-style AI-news app for niche AI/tech categories. Content is researched and fully generated by an external personal AI agent (**OpenClaw**), reviewed by a human admin in the **Admin Dashboard**, and consumed anonymously by readers in the **News App**. Brand: **Armal News**.

## MVP scope (locked)

**In scope:**

- **English only.** No Hindi.
- **No TTS / audio.** No karaoke read-along.
- A **Story** has exactly: **Title**, **Image**, **Short Summary** (catchy, card-sized), **Full Article Text**, **Source Link**, one or more **Categories** (from a fixed seeded list), and zero or more **Tags** (free-form text).
- **Card view**: Title + Image + Short Summary, full-screen vertical-snap (TikTok/Reels) on mobile.
- **Floating overlays on the feed** (low-opacity over the image):
  - **Armal News logo** in the top-center. Tap → feed scrolls to story #1 of the current category.
  - **Category icon** in the top-left. Tap → bottom sheet slides up with the **All**/category list. Currently-selected category shows as a pill next to the icon ("AI in Tech ✕"). Tap a category → sheet closes, feed reloads filtered to that category from story #1.
- **Deep dive view**: tap a card → `/story/[slug]` page → Image on top, then Title, then Full Article Text, then "View source" link.
- **Categories filter** on the feed: feed defaults to **All** (every published story). User can select one Category at a time to filter; the feed then shows only stories tagged with that Category. Categories are a fixed admin-seeded list (see below); they are not user-creatable. Each Story can belong to multiple Categories.
- **Tags** on every Story (free-form, generated per-story by OpenClaw). Tags are not used for primary navigation in MVP, but are stored and surfaced in the deep-dive view.
- **Theme switcher**: light + dark; user-toggleable; respects OS preference by default.
- **SSR'd shareable URLs**: `/story/[slug]` rendered server-side with full Open Graph metadata. The launch strategy depends on these being shareable.
- **Native share** on mobile (`expo-sharing`).
- **Source-link click-out** at the bottom of every full-article view.
- **PWA installable on desktop browsers only.** Mobile browser visitors are routed to App Store / Play Store install links via a banner — they should not install the website as a PWA on their phone; they should install the real app.

**Out of scope for MVP** (deferred to "next version" or later):

- AI Journey Timeline (separate page mapping AI history milestones).
- User accounts, Google OAuth, Profile, Likes/Saved.
- Onboarding / category-selection flow.
- Hindi translation, TTS audio, karaoke read-along.
- Author personas, read-time estimates.
- In-feed ads, affiliate-link highlighting.
- Premium tier features (Prompt Generator, Verify-this-Story, Deep Analysis, AI Tools Database).
- Editor Agent / automated QA via Claude Agent SDK pre-publish (admin reviews everything by hand at MVP cadence).

## Story lifecycle

1. **OpenClaw** researches and generates a batch of fully-formed Stories (title + summary + body + image + source link + category) on the Linux laptop.
2. OpenClaw POSTs the batch to the **Admin Dashboard** at `localhost:3001` over Tailscale. Each Story lands in Postgres as `status='draft'`.
3. The **Admin** opens the Admin Dashboard, reviews each draft, optionally edits text, and clicks **Publish** (status → `published`) or **Reject** (status → `rejected`).
4. The **News App** (web + mobile) immediately serves any Story with `status='published'`.

## Language

**OpenClaw**:
An external personal AI assistant (https://github.com/openclaw/openclaw) tasked by the admin to perform news research and produce **fully-formed Stories** (title, short summary, full article, image) in the schema this project specifies. Lives outside this codebase. The current `ai_news_feed_comprehensive.json` is a probe of OpenClaw's capability, not the final schema.
_Avoid_: Content Engine, News Researcher, OpenClaw Pipeline, OpenClaw Engine.

**Story**:
The single content unit, fully-formed by OpenClaw. Has: a **Title**, an **Image**, a **Short Summary** (catchy, card-sized; rendered in serif italic + curly quotes), a **Full Article Text** (Markdown, deep-dive body), a source **Link**, one or more **Categories** (from the seeded list), zero or more free-form **Tags**, and an **External ID** (OpenClaw-supplied, used as idempotency key on re-delivery). A Story moves through states: `draft` (awaiting admin review) → `published` (visible to readers) — or `rejected` (soft-deleted, retained for audit). Drafts can be edited freely; published stories can also be edited in place (typo fixes, link corrections); the slug never changes once assigned.

**Category**:
A niche bucket from a **fixed, admin-seeded list** (e.g. _AI in Tech_, _AI in Finance_, _AI in Robotics_). Each **Story** belongs to **one or more** Categories. Readers can filter the feed by selecting a single Category at a time (or **All**, which is the default). Categories are not user-creatable. _Avoid_: Niche, Topic, Bucket.

**Tag**:
A free-form text label attached to a **Story** (e.g. "humanoid-robot", "1x", "frontier-model"). Tags are produced by OpenClaw per Story; there is no central Tag table — they are stored as a string array on the Story row. Tags are surfaced in the deep-dive view but are not primary feed navigation in MVP. _Avoid_: Keyword, Label.

**Admin Dashboard**:
The central internal web app. Single source of truth for content lifecycle. OpenClaw delivers raw output here; the admin reviews and approves stories here; approved stories are published from here to the **News App**.
_Avoid_: OpenClaw Dashboard, Admin Platform, Internal Tool.

**News App**:
The public-facing app readers consume — two surfaces sharing one content pipeline:
- **News App (web)**: Next.js, server-side-rendered for SEO and social sharing.
- **News App (mobile)**: React Native + Expo, shipped to iOS App Store and Google Play.

Both surfaces read the same content from Postgres. The mobile surface goes through Next.js API routes hosted by **News App (web)**; the web surface reads directly via SSR. Neither speaks to **OpenClaw**. See ADR-0002.
_Avoid_: Public App, Main Site, Frontend.

**Admin**:
The single human (you) operating the **Admin Dashboard**. Approves or rejects each story before it becomes public.

**Reader**:
An end user of the **News App**. May be anonymous or signed-in.

## Relationships

- **OpenClaw** → **Admin Dashboard**: OpenClaw delivers fully-formed Stories (JSON + images).
- **Admin** ↔ **Admin Dashboard**: Admin reviews each Story and flips its status from `draft` to `published` (or rejects it).
- **Admin Dashboard** ↔ **News App**: Both apps share a single Postgres database. The Admin Dashboard writes Stories; the News App reads only Stories where `status = 'published'`. There is no HTTP publish call, no file copy, no webhook between them — publication is a state flip in shared storage.
- **Reader** ↔ **News App**: Readers browse the feed **anonymously**. There are no user accounts, no Likes, no Profile, no onboarding in MVP.

## Topology (where things run)

- **OpenClaw** runs on the admin's **Linux laptop** as a personal AI agent. Not deployed.
- **Admin Dashboard** runs on the admin's **MacBook** at `localhost:3001` (Next.js). Not deployed. The MacBook can sleep at any time without affecting readers.
- **OpenClaw → Admin Dashboard**: HTTPS POST over **Tailscale**'s private mesh (no public internet exposure, no shared-secret auth needed — Tailscale handles peer auth).
- **Postgres**: hosted on **Supabase** (cloud). Both the Admin Dashboard (laptop) and the News App (deployed) connect via `DATABASE_URL`. See ADR-0003.
- **Image storage**: **Supabase Storage** bucket. OpenClaw uploads (via the Admin Dashboard ingest endpoint) and the News App reads via Supabase's CDN.
- **News App (web)**: Next.js, deployed (host TBD).
- **News App (mobile)**: React Native + Expo, distributed via Apple App Store and Google Play.

> **Tooling note:** The Supabase MCP server is intentionally **not** used for this project — it's currently connected to a different Supabase project. See `CLAUDE.md`.

**Typography:**

- **Newsreader** (Google Fonts, free) — headings, story titles, italic quoted summary, wordmark.
- **Inter** (Google Fonts, free) — body, full article text, UI labels.
- **JetBrains Mono** (Google Fonts, free) — code blocks inside article markdown.

**Card composition** (one full-screen vertical-snap card):

- **Image**: top 1/3 of the card height, full bleed.
- **Headline overlay box**: centered, midpoint sits on the image's bottom edge so it overlaps both. Surface color fill, 1px hairline border, 24px padding, 12px radius. Newsreader 600, ~24px, primary color, left-aligned, max 2 lines.
- **Summary panel**: bottom 2/3, surface color. Summary in Newsreader **italic** ~20px, line-height 1.5, wrapped in typographic curly quotes (`"…"`), centered vertically, 32–48px horizontal padding.
- **"Tap to read →"** affordance bottom-right, muted text color.
- **Floating overlays** on top of the card (low-opacity): category icon top-left (24px, ~50% opacity), **Armal News** wordmark top-center (Newsreader italic, ~16px, ~60% opacity, all-caps tracking +0.05em). Tap overlays = the locked behaviors (icon → category sheet, wordmark → scroll to story #1).

**Brand mark for MVP:** wordmark only. App-store icon = stylized "A" in Newsreader on warm-paper background. Real logo deferred to post-MVP.

## Tech stack (locked)

- **Monorepo**: pnpm workspaces + Turborepo. Apps in `apps/{web,mobile,admin}`; shared code in `packages/shared` (zod schemas, types, constants).
- **Backend / API**: Next.js API routes inside `apps/web` (no separate `apps/api`). The Admin Dashboard owns its own write endpoints.
- **ORM / DB**: **Drizzle ORM** against Postgres (Supabase). Schema in `packages/shared/db` so all three apps can import types; migrations as plain SQL via `drizzle-kit`.
- **Web News App**: Next.js (App Router), React Server Components for SSR'd `/story/[slug]`, Tailwind + dark mode, installable as a desktop PWA.
- **Mobile News App**: React Native + Expo (managed workflow). Distributed via App Store + Play Store.
- **Admin Dashboard**: Next.js, runs at `0.0.0.0:3001` on the admin's MacBook (so OpenClaw can reach it over Tailscale). **No app-level authentication in MVP** — Tailscale is the security boundary.
- **OpenClaw**: external personal AI agent on the admin's Linux laptop, posts batches to the Admin Dashboard over Tailscale.
- **Image storage**: Supabase Storage bucket; served via its CDN.
- **Hosting**: Web News App on **Vercel** (Next.js native, OG-image generation built-in, free hobby tier sufficient for MVP); Admin Dashboard runs on the admin's laptop and is intentionally not deployed in MVP.
- **Story body format**: **CommonMark Markdown**, sanitized at ingest. Web renders via `react-markdown`; mobile renders via `react-native-markdown-display`.
- **Image strategy**: OpenClaw produces one hi-res portrait image per story (~1200×1600px, ≤2 MB JPEG/PNG/WebP). Stored as-is in Supabase Storage. Web optimizes via Next `<Image>`; mobile via `expo-image`. OG previews generated dynamically by Next's `opengraph-image.tsx`.

> **Migration plan:** Once revenue justifies it, OpenClaw and the Admin Dashboard will be moved off the admin's laptops onto cloud hosting with proper auth. The schema, API contracts, and code paths do not change for that migration — only deployment topology.
