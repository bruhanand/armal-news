# PRD 0001 — Armal News MVP

**Status:** needs-triage
**Owner:** Admin (solo)
**Source-of-truth references:** `CONTEXT.md`, `docs/adr/0001`, `docs/adr/0002`, `docs/adr/0003`. Where `PRODUCT_BLUEPRINT.md` / `PARTNER_PITCH.md` / `IMPLEMENTATION_GUIDE.md` / `IMPLEMENTATION_PLAN.md` conflict with those, the latter win.

## Problem Statement

As a Reader interested in niche AI/tech topics, I am drowning in long-form, generic AI news on the open web. Existing apps either bury me in dense articles or fragment the topic across dozens of feeds. I want a way to skim what just happened in the corners of AI I care about (AI in robotics, AI in finance, AI in cooking, etc.) at the speed of a TikTok feed, then dive deeper on the few stories that actually matter to me — and share them with friends without forcing them to install an app first.

As the Admin, I have a personal AI agent (**OpenClaw**) on my Linux laptop that already produces fully-formed Stories (title + body + image + source). I need a place to (a) receive batches from OpenClaw without exposing anything to the public internet, (b) review each story by hand and either publish or reject it, and (c) have published stories appear immediately in a public reader-facing app on the App Store, Play Store, and the web — without operating a server fleet just for the admin tool.

## Solution

**Armal News** — a TikTok-style, full-screen vertical-snap feed of curated AI news Stories, available as a Next.js web app (Vercel-hosted, SSR'd, desktop-PWA-installable, mobile-routes-to-app-store) and a React Native + Expo mobile app on iOS and Android.

A **Story** is the single content unit, fully-formed by OpenClaw: Title, Image, Short Summary (catchy, card-sized, rendered in Newsreader italic with curly quotes), Full Article Text (CommonMark Markdown), Source Link, one or more Categories from a fixed seeded list of nine, zero-or-more free-form Tags, and an External ID used as idempotency key on re-delivery. Each story moves through `draft` → `published` (or `rejected`).

OpenClaw POSTs batches to the **Admin Dashboard** (Next.js, runs at `0.0.0.0:3001` on the admin's MacBook) over **Tailscale** — no public internet exposure, no shared-secret auth (Tailscale is the security boundary). The Admin reviews each draft, optionally edits the text, and clicks **Publish** or **Reject**. Publication is a state flip on the `status` column of a Postgres row shared by both apps (ADR-0001) — there is no HTTP publish call.

Readers consume the feed anonymously. The feed defaults to **All**; tapping the category icon opens a bottom sheet with the nine categories; tapping one filters the feed to that category from story #1. Tapping a card opens a deep-dive at `/story/[slug]` (SSR'd with OG metadata for sharing). The Armal News wordmark in the top-center scrolls back to story #1 of the current filter.

Postgres and image storage are on **Supabase** (cloud) so readers continue to be served when the admin's MacBook sleeps (ADR-0003). The Admin Dashboard is intentionally **not** deployed in MVP.

## User Stories

### Reader — feed (web + mobile)

1. As a Reader, I want to open the app/site and immediately see the most recent published Story as a full-screen card, so that I can start consuming without onboarding, account creation, or category-selection.
2. As a Reader, I want a vertical-snap gesture to advance to the next Story, so that the rhythm matches TikTok / Instagram Reels and feels natural on a phone.
3. As a Reader on the web, I want the same vertical-snap behavior in the browser as on mobile, so that the experience is consistent across surfaces.
4. As a Reader, I want each card to show the Story's image (top third), Title (overlaid headline box at the seam), and Short Summary in italic serif with curly quotes (bottom two-thirds), so that I can decide in three seconds whether to dive deeper.
5. As a Reader, I want a "Tap to read →" affordance bottom-right of every card, so that I know the card is interactive.
6. As a Reader, I want the Armal News wordmark floating top-center on every card at low opacity, so that the brand is present without being intrusive.
7. As a Reader, I want tapping the wordmark to scroll the feed back to story #1 of my current filter, so that I have a one-tap "go to top" gesture.
8. As a Reader, I want a category icon top-left on every card at low opacity, so that I can see and change the current category filter without leaving the feed.
9. As a Reader, I want tapping the category icon to slide up a bottom sheet listing **All** plus the nine seeded categories, so that I can pick a niche.
10. As a Reader, I want the currently-selected category to appear as a dismissible pill next to the category icon (e.g., "AI in Tech ✕"), so that the active filter is always visible at a glance.
11. As a Reader, I want tapping a category in the sheet to close the sheet and reload the feed filtered to that category from story #1, so that switching context is one gesture.
12. As a Reader, I want the **All** option in the category sheet to show every published Story in reverse chronological order, so that I have a default "everything" view.
13. As a Reader, I want stories tagged with multiple categories to appear under each of those categories, so that no story is hidden when I narrow my filter.
14. As a Reader on mobile, I want infinite-scroll pagination of the feed, so that I can keep swiping until I've seen everything published.
15. As a Reader, I want my position in the feed to NOT be remembered across sessions in MVP, so that opening the app always lands me on the freshest story.

### Reader — deep dive

16. As a Reader, I want tapping a card to open `/story/[slug]` showing the Image on top, then the Title, then the Full Article Text rendered from Markdown, then a "View source" link to the original URL, so that I can read the long-form treatment.
17. As a Reader, I want the deep-dive page to surface the Story's Tags (e.g., "humanoid-robot", "1x"), so that I can see at a glance what entities the story is about, even though I cannot filter by tag in MVP.
18. As a Reader, I want the deep-dive page to be server-rendered with full Open Graph metadata (og:title, og:description from the Short Summary, og:image), so that pasting the URL into iMessage, Twitter, WhatsApp, or Slack produces a rich preview card.
19. As a Reader on mobile, I want a native share sheet (`expo-sharing`) on the deep-dive page that shares the Story's `/story/[slug]` URL, so that the recipient sees a real preview rather than a generic link.
20. As a Reader, I want code blocks inside article markdown to render in JetBrains Mono, so that they're visually distinct from prose.
21. As a Reader, I want the source-link click-out to be obvious at the bottom of every full-article view, so that I can attribute / verify before I share.

### Reader — theme + chrome

22. As a Reader, I want the app/site to follow my OS dark/light preference by default, so that I don't have to configure anything.
23. As a Reader, I want a manual theme toggle that overrides the OS preference and persists, so that I can read in dark mode at noon if I want to.
24. As a Reader on a desktop browser, I want a subtle "Install Armal News" prompt that turns the site into a desktop PWA, so that I can launch it from my dock.
25. As a Reader on a mobile browser, I want a banner that routes me to the App Store or Play Store install link rather than offering a PWA install, so that I get the real native app and not a webview wrapper on my phone.

### Reader — content quality (driven by ingest)

26. As a Reader, I want every Story to have an image, a non-empty short summary, and a non-empty body, so that no card ever shows broken content (the ingest layer rejects stories missing any of these).
27. As a Reader, I never want to see `draft` or `rejected` Stories in the feed, the deep-dive, the API, or share previews, so that the public surface is a strict subset of `status='published'`.

### Admin — receiving from OpenClaw

28. As the Admin, I want OpenClaw on my Linux laptop to POST a batch of fully-formed Stories to my Admin Dashboard at `localhost:3001` over Tailscale, so that there's no public webhook to harden and no shared secret to rotate.
29. As the Admin, I want each Story in the OpenClaw payload to carry an **External ID** (OpenClaw-supplied) that the ingest endpoint uses as an idempotency key, so that re-delivering a batch on retry doesn't create duplicates — it updates the existing draft in place.
30. As the Admin, I want the ingest endpoint to validate every incoming Story against a strict zod schema (title, image URL or binary, short summary, body, source link, ≥1 category from the fixed list, tags array, external_id) and reject the entire batch if any single Story fails validation, so that bad output from OpenClaw never silently lands in my queue.
31. As the Admin, I want the ingest endpoint to download the image referenced by OpenClaw, store it as-is in a Supabase Storage bucket, and persist the resulting CDN URL on the Story row, so that the public News App is not depending on whichever upstream URL OpenClaw originally scraped.
32. As the Admin, I want the ingest endpoint to sanitize the Markdown body at write time (strip disallowed HTML, cap heading levels, etc.), so that the rendered output cannot inject script tags or break layout.
33. As the Admin, I want every newly-ingested Story to land with `status='draft'`, regardless of what OpenClaw says, so that I am the single gate to public visibility.
34. As the Admin, I want the ingest endpoint to assign a slug at draft creation time and freeze it for the Story's lifetime, so that the public URL never changes after publication, even if I edit the title.

### Admin — review queue

35. As the Admin, I want to open the Admin Dashboard at `localhost:3001` from my MacBook and see a queue of all `draft` Stories ordered by ingest time, so that I can work through the backlog newest- or oldest-first.
36. As the Admin, I want to click into any draft to see its image, title, short summary, full article body (rendered as Markdown), source link, categories, tags, and external ID, so that I can review every visible field at once.
37. As the Admin, I want to inline-edit any text field on a draft (title, short summary, body, source link, tags, categories), so that I can fix typos and OpenClaw mistakes without rejecting the whole story.
38. As the Admin, I want to inline-edit the categories on a draft, picking one or more from the seeded list of nine, so that I can correct OpenClaw's classification.
39. As the Admin, I want a **Publish** button on the draft detail view that flips `status` from `draft` to `published` and stamps `published_at`, so that the Story appears in the public News App immediately.
40. As the Admin, I want a **Reject** button that flips `status` to `rejected` (soft-delete; the row is retained for audit) and removes the story from my queue, so that I can clear out content I don't want without losing the audit trail.
41. As the Admin, I want to be able to also edit a Story after it's published (typo fixes, link corrections), with the slug pinned, so that I can correct mistakes without rotating share URLs.
42. As the Admin, I want the dashboard to operate without app-level authentication in MVP — Tailscale is the security boundary (ADR-0003) — so that I don't have to maintain an admin auth provider for a single-user tool.
43. As the Admin, I want the Admin Dashboard to NOT be deployed publicly in MVP (it runs only on my MacBook), so that there is no internet-exposed admin endpoint to attack.

### Cross-cutting — content pipeline

44. As the Admin, I want the Admin Dashboard and the public News App to share a single Postgres database (ADR-0001), so that publication is a state flip — never an HTTP publish call, never a file copy, never a webhook.
45. As the Admin, I want both apps to evolve schema in lock-step via the same Drizzle migrations, so that I never operate with version skew between writer and reader.
46. As the Admin, I want Postgres and image storage to be cloud-hosted on Supabase (ADR-0003), so that readers see the public News App 24/7 even when my MacBook is asleep.
47. As the Admin, I want all three apps (web, mobile, admin) to import their zod schemas, types, and DB schema from a single `packages/shared` workspace, so that a schema change is a one-line edit applied everywhere.

### Cross-cutting — categories + tags

48. As the Admin, I want the Category list to be a fixed, admin-seeded list of nine entries (`ai-in-tech`, `ai-in-finance`, `ai-in-healthcare`, `ai-in-robotics`, `ai-in-cooking`, `ai-in-education`, `ai-research`, `ai-tools`, `ai-policy-safety`), so that the navigation never sprawls and is not user-creatable.
49. As the Admin, I want the seed migration to upsert the nine categories on every fresh-DB bootstrap, so that local dev / staging / prod have identical category sets.
50. As the Admin, I want each Story to belong to one-or-more Categories via a join table, not a single foreign key, so that "AI in Robotics" and "AI in Tech" can both surface the same story.
51. As the Admin, I want Tags to be stored as a free-form text array on the Story row (no central tag table), since they are produced by OpenClaw per-Story and are not used for primary navigation in MVP.

### Sharing + SEO

52. As a Reader who pastes a Story URL into Twitter / iMessage / WhatsApp / Slack, I want to see a rich preview card with the Story's image, title, and short summary, so that the recipient understands what they're being sent before clicking.
53. As the Admin, I want the web Story pages to be SSR'd via the Next.js App Router with React Server Components, so that crawlers (Google, Twitterbot, Slackbot, etc.) see fully-rendered HTML and OG metadata on first byte.
54. As the Admin, I want OG images generated dynamically by Next's `opengraph-image.tsx` at the per-Story level, so that I never have to author OG cards by hand.

## Implementation Decisions

### Modules

**`packages/shared`** (imported by `apps/web`, `apps/mobile`, `apps/admin`):

- **`db/schema`** — Drizzle table definitions: `stories` (id, external_id unique, slug unique, title, short_summary, body_markdown, image_url, source_link, status enum [`draft` | `published` | `rejected`], created_at, updated_at, published_at nullable, tags text[]); `categories` (id, slug unique, name, sort_order); `story_categories` join (story_id, category_id, composite PK).
- **`db/queries`** — Read helpers used by both web SSR and admin: `listPublishedStories({ category?, cursor?, limit })`, `getPublishedStoryBySlug(slug)`, `listCategories()`. Each filters on `status='published'` for reader-facing calls; admin reads have a separate `listDraftStories` helper.
- **`db/migrations`** — Plain SQL files generated by `drizzle-kit`, including a category-seed migration that upserts the nine MVP entries.
- **`validation/story`** — zod schemas: `IngestStoryV1` (the OpenClaw payload), `EditStoryPatch`, `StatusTransition` (enforces `draft → published | rejected`; published-in-place edits keep `published`).
- **`lib/slugify`** — Pure function: title → URL-safe slug; on collision appends `-2`, `-3`, ….
- **`lib/markdown-sanitize`** — Pure function: CommonMark in, sanitized CommonMark out (allow-list of safe HTML, capped heading depth, no raw `<script>` / `<iframe>` / event handlers).
- **`design/tokens`** — Color and type tokens from `CONTEXT.md`, exported as both Tailwind CSS variables (consumed by web + admin) and a TS object (consumed by RN).
- **`constants/categories`** — Single source of truth for the nine seeded categories (slug, display name, icon key); the seed migration imports from here so the seed and the runtime list never drift.

**`apps/admin`** (Next.js, runs on `0.0.0.0:3001` on admin's MacBook, no app-level auth):

- **`POST /api/ingest/stories`** — Accepts a batch of Stories from OpenClaw. Pipeline: zod-validate the full batch → fetch each image and upload to Supabase Storage → sanitize markdown → upsert each Story by `external_id` (insert if new, update fields if known) with `status='draft'` and a frozen slug. Returns per-Story status to OpenClaw so it can log delivery success.
- **`PATCH /api/admin/stories/[id]`** — Inline edit endpoint for any field except slug.
- **`POST /api/admin/stories/[id]/publish`** — Status transition `draft → published` (or no-op if already `published`); stamps `published_at` on first transition.
- **`POST /api/admin/stories/[id]/reject`** — Status transition → `rejected` (soft-delete; row retained).
- **Admin UI** — Drafts queue page (table of drafts, newest first, one-click open) and draft detail page (editable form with Publish / Reject buttons). React Server Components on the read paths; server actions for the write paths.

**`apps/web`** (Next.js App Router, deployed to Vercel):

- **`GET /api/feed?category=&cursor=&limit=`** — Reader-facing feed. Strictly filters `status='published'`. Optional category filter (slug); pagination via cursor on `published_at`.
- **`GET /api/story/[slug]`** — Single-story fetch. Returns 404 if status ≠ `published`. Used by the mobile app and as a fallback by the web client.
- **`/`** — Feed shell. RSC fetches the first page server-side; a client component owns the vertical-snap interaction, the floating overlays (wordmark, category icon, pill), the bottom-sheet category picker, and pagination.
- **`/story/[slug]`** — SSR'd deep-dive page (image, title, body, tags, source link). Includes route-segment `opengraph-image.tsx` that renders the OG card dynamically from Story fields.
- **PWA manifest + install banner** — Desktop browsers can install as a PWA; mobile-UA detection routes to App Store / Play Store deep links instead. Critical: the mobile web experience must NOT offer PWA install — readers on phones should land on the real native app.
- **Theme provider** — Defaults to OS preference; manual toggle persists in `localStorage`.

**`apps/mobile`** (React Native + Expo, managed workflow, App Store + Play Store):

- Feed screen using a virtualized list (FlashList) with full-screen vertical paging.
- Deep-dive screen rendering `react-native-markdown-display`.
- Category bottom sheet with the same nine categories.
- Native share via `expo-sharing` (shares the `https://…/story/[slug]` URL so recipients get the SSR'd preview).
- Theme provider with AsyncStorage persistence.
- Typed API client targeting `apps/web`'s public routes (`/api/feed`, `/api/story/[slug]`).

### API contracts

- **OpenClaw → Admin Dashboard** (`POST /api/ingest/stories`): batch envelope `{ stories: IngestStoryV1[] }` where each Story has `external_id`, `title`, `short_summary`, `body_markdown`, `image_url` (or base64 inline — TBD; default to URL), `source_link`, `category_slugs: string[]` (must all match seeded slugs), `tags: string[]`. Idempotent on `external_id`.
- **Reader → Web News App** (`GET /api/feed`, `GET /api/story/[slug]`): JSON, filtered to `status='published'` only. Slugs are stable for life of Story.

### Schema

See `packages/shared/db/schema` description above. Key invariants:

- `external_id` is unique; it's the idempotency key for ingest.
- `slug` is unique and immutable after first assignment.
- `status` is a Postgres enum; the only legal transitions are `draft → published`, `draft → rejected`, `published → published` (in-place edit).
- `published_at` is nullable; first set on `draft → published`; never cleared.
- `story_categories` enforces ≥1 category at the application layer (zod), not via a DB constraint.

### Architectural decisions already made (referenced, not re-litigated here)

- **ADR-0001**: Admin Dashboard and News App share one Postgres; publication is a state flip.
- **ADR-0002**: Mobile = React Native + Expo; web = Next.js. Both consume the same HTTP API and shared Postgres.
- **ADR-0003**: Admin Dashboard runs on the admin's MacBook over Tailscale; Postgres is on Supabase so readers are unaffected by the laptop sleeping. No app-level admin auth in MVP.

### Tooling notes (from `CLAUDE.md`)

- The Supabase MCP server is currently connected to a different project. Database migrations are managed by hand via `drizzle-kit`-generated SQL files run with the `supabase` CLI / `psql`. Do not call `mcp__claude_ai_Supabase__*` until the user gives explicit clearance.

## Testing Decisions

A good test in this codebase exercises **external behavior** through a stable interface — what a caller sees, not how the module is structured internally. Tests should not assert on private helpers, internal field names, or call-counts on collaborators that aren't part of the contract. Snapshot tests are reserved for Markdown / OG-image rendering where the output IS the contract.

There is no prior art in this repo (greenfield). The conventions established by the first batch of tests should be:

- **Pure-function tests** (Vitest in Node) for `slugify`, `markdown-sanitize`, the zod ingest validator, and the status-transition guard. Fast, deterministic, no I/O.
- **Repository tests** (Vitest against a local Postgres started by `supabase start` or a Dockerized Postgres) for `db/queries` — assert on what the SQL returns given a seeded fixture, not on the SQL string.
- **Ingest pipeline tests** (Vitest with mocked Supabase Storage) for `POST /api/ingest/stories` — assert the end-to-end behavior: a valid batch yields N draft rows with correct slugs, sanitized bodies, and stored image URLs; an invalid batch yields zero rows and a useful error.
- **Status-transition tests** for publish / reject endpoints — assert legal transitions work and illegal ones are rejected.

Modules to write tests for in MVP:

- `lib/slugify` — pure
- `lib/markdown-sanitize` — pure (snapshot-test the rendered output for a curated set of malicious + benign inputs)
- `validation/story` — zod schemas, especially the category-slug must-be-in-seeded-list rule
- Status-transition guard — pure
- `db/queries.listPublishedStories` — must filter `status='published'`, must support category filter, must order by `published_at DESC`
- `db/queries.getPublishedStoryBySlug` — must 404 on `draft` / `rejected`
- Ingest orchestration — idempotency on `external_id`, image upload, markdown sanitization, slug freezing
- Publish / reject endpoint behavior

UI is **not** unit-tested in MVP; the admin verifies feed and admin behavior in the browser before each cut. Add component tests (Playwright or Vitest + Testing Library) post-MVP if regressions become a recurring problem.

## Out of Scope

The following items are explicitly deferred (per `CONTEXT.md` "Out of scope for MVP"):

- AI Journey Timeline page.
- User accounts, Google OAuth, Profile, Likes, Saved.
- Onboarding flow / user-driven category selection.
- Hindi translation, TTS audio, karaoke read-along.
- Author personas, read-time estimates.
- In-feed ads, affiliate-link highlighting.
- Premium tier (Prompt Generator, Verify-this-Story, Deep Analysis, AI Tools Database).
- Editor Agent / automated QA via Claude Agent SDK pre-publish — admin reviews everything by hand at MVP cadence.
- Tag-based filtering or a central Tag table — Tags are surfaced on deep-dive but are not navigation.
- App-level auth on the Admin Dashboard — Tailscale is the security boundary.
- Public deployment of the Admin Dashboard — it runs on the admin's MacBook only.
- Reader feed-position memory across sessions.
- Mobile PWA install — mobile browsers route to the App Store / Play Store instead.
- A separate `apps/api` workspace — Next.js API routes inside `apps/web` and `apps/admin` cover all server-side needs in MVP.

## Further Notes

- **Migration trigger** (from ADR-0003): once revenue justifies the ops cost, the Admin Dashboard and OpenClaw move to cloud hosting and an auth provider gets bolted on. Schema, API contracts, and code paths do not change — only deployment topology. Keep that in mind when designing module boundaries; nothing should assume "this runs on the admin's laptop" beyond the Tailscale-reachable hostname in env config.
- **Schema evolution risk** (from ADR-0001): a breaking schema change requires deploying both apps together. Prefer additive migrations; cut feature flags rather than columns when possible.
- **OpenClaw probe vs. final schema**: the current `ai_news_feed_comprehensive.json` only contains `title`, `link`, and `article`. The MVP schema additionally requires `short_summary`, `image`, `external_id`, ≥1 `category_slug`, and `tags[]`. OpenClaw's output contract has to expand to match this PRD before the ingest endpoint will accept its batches; aligning OpenClaw on the `IngestStoryV1` zod schema is a prerequisite to first end-to-end run.
- **Brand mark**: MVP uses the wordmark only. The App Store icon is a stylized "A" in Newsreader on the warm-paper background. Real logo work is post-MVP.
- **Hosting**: Web News App on Vercel (free hobby tier sufficient for MVP). Admin Dashboard intentionally not deployed.
